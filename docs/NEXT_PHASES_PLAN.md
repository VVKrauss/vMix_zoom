# План следующих фаз (главная, авторизация, презентации)

Этот файл — **единая точка входа**: когда говорите «приступаем к следующим фазам» или «смотри NEXT_PHASES», ориентируемся на разделы ниже по порядку или выборочно, как договоримся.

## Фаза 1 — Отдельное окно (страница) для главной

**Цель:** вынести «домашний» сценарий из текущего потока (сейчас при `status !== 'connected'` показывается `JoinPage`) в полноценную **главную страницу** как отдельный маршрут/экран.

### Задачи (черновик)

- Ввести **маршрутизацию** (например, `react-router`: `/` — главная, `/room/:roomId` или `/join` — вход в комнату, при необходимости редиректы).
- **Главная (`/`)**: брендинг, краткое описание, кнопка «Создать комнату» / «Войти по ссылке», возможно список недавних комнат (локально в `localStorage` — опционально).
- **Join** остаётся формой имени + комнаты + пресета, но живёт на отдельном URL или в модалке с главной — решение при реализации.
- Согласовать **историю URL** с уже используемым `replaceRoomInBrowserUrl` и соло-режимом (`?peer=`), чтобы не сломать deep link’и.

### Риски / вопросы

- Нужен ли **SSR** или достаточно SPA-only.
- Язык и контент главной (статический текст vs CMS позже).

### Критерий готовности

- Пользователь открывает сайт и видит отдельную главную; переход в комнату — осознанный шаг с понятным URL.

### URL и идентификатор комнаты (уточнение)

- **Не по «названию» в URL** — в пути только **стабильный технический id** (slug): короткий **nanoid** (8–12 символов) или **UUID**. Человекочитаемое имя комнаты (если понадобится) — отдельное поле в UI/API, не часть пути.
- **Рекомендуемая схема путей (SPA):**
  - `/` — главная: только вход в продукт («Создать комнату» / «У меня есть код»).
  - `/r/:roomId` — страница комнаты: форма имени + кнопка «Войти» (пока не подключены) и дальше текущий `RoomPage` после `join`.
- **Соло-зритель** сейчас: `?room=…&peer=…`. После перехода на путь удобно: **`/r/:roomId?peer=…`** (peer остаётся в query). В коде нужно брать `roomId` из **параметра маршрута**, а не только из `?room=`. Переходный период: поддержать и старый `/?room=&peer=` (редирект на `/r/...`).
- **«Создать комнату»:** клиент генерирует новый `roomId` (nanoid), делает `navigate('/r/' + id)` — пользователь вводит имя и жмёт «Войти» → тот же `joinRoom`, что сейчас.
- **«Войти по id»:** поле на главной → `navigate('/r/' + normalize(id))` (обрезка пробелов, нижний регистр по желанию).

### Что нужно на бэкенде (room-platform)

Сейчас фронт шлёт **`joinRoom`** с `{ roomId, name }`. Типичное поведение демо-mediasoup: **комната создаётся при первом join**, если её ещё нет.

| Сценарий | Нужны ли изменения на бэке |
|----------|----------------------------|
| «Создать комнату» = новый случайный `roomId` + первый участник делает `joinRoom` | **Часто нет** — если сервер уже создаёт комнату по `roomId` при join. |
| Гарантировать, что id **уникален** и не пересечётся с чужим случайным выбором | Либо длинный UUID/nanoid на клиенте, либо эндпоинт **`createRoom` → { roomId }** на сервере. |
| Запретить join в несуществующую комнату (только по приглашению) | **Да:** отдельное создание комнаты + `joinRoom` с ошибкой `ROOM_NOT_FOUND`. |
| Лимиты, TTL пустых комнат, модерация | Отдельная логика на сервере, контракт ошибок для фронта. |

**Итог:** минимальный фронт под «своя страница на комнату» **может обойтись без доработки бэка**, если остаётся модель «любой `roomId` при первом join создаёт комнату». Для продуктового поведения («только существующие комнаты») понадится **`createRoom`** и/или проверка существования до join.

## Фаза 2 — Авторизация

**Цель:** идентифицировать пользователя до/после входа в комнату, чтобы ограничивать доступ, персонализировать имя и (позже) хранить данные на сервере.

### Подфазы (на выбор)

1. **Минимум:** гостевой вход с отображаемым именём (как сейчас) + опционально «запомнить имя» в `localStorage` (уже частично есть идеи в проекте).
2. **Полноценный аккаунт:** регистрация / вход (email+пароль, magic link, OAuth — выбор стека).
3. **Связка с room-platform:** выдача **JWT** или сессии, передача токена в Socket.IO / joinRoom, проверка на сервере.

### Задачи (черновик)

- Выбрать **провайдера** (самописный backend, Supabase, Clerk, Keycloak и т.д.) и модель пользователя (`id`, `email`, `displayName`).
- Защищённые маршруты: главная публичная, комната — по политике (публичная ссылка vs только для залогиненных).
- UI: формы входа/регистрации, состояние «загрузка / ошибка», выход.
- Обновить **документацию** контрактов с бэкендом (какие поля уходит в `joinRoom` вместе с токеном).

### Риски / вопросы

- Публичные комнаты без аккаунта — остаются или уходят.
- GDPR / хранение PII, если email обязателен.

### Критерий готовности

- Один согласованный способ «кто я» для клиента и сервера; комнаты можно открывать с учётом правил доступа.

## Фаза 3 — Презентации (PPTX / PDF) как «демонстрация» с управлением

**Цель:** загрузить файл **PPTX** или **PDF** и показывать его участникам **как отдельный источник демонстрации** (не захват всего экрана), с **элементами управления** (листание слайдов, полноэкранный режим у зрителя, пауза и т.д. — уточняется).

### Подходы (на высоком уровне)

| Формат | Варианты реализации | Плюсы / минусы |
|--------|---------------------|----------------|
| **PDF** | Рендер страниц в `<canvas>` / `pdf.js`, трансляция кадра или отдельных страниц как видео/изображения | Проще контролировать «слайды»; нужен пайплайн в WebRTC (см. ниже). |
| **PPTX** | Конвертация на **сервере** в PDF или набор изображений; либо клиентская библиотека (тяжело, ограниченно) | Надёжнее через сервер; без бэкенда — очень ограниченно. |
| **Трансляция** | Текущий путь **screen share** заменить/дополнить **виртуальным MediaStream** (`canvas.captureStream()` + тайминг) или отдельным video track с картинками | Вписывается в mediasoup как video producer с `appData: { source: 'presentation' }`. |

### Задачи (черновик)

- **UI:** зона загрузки файла, превью, панель «следующий / предыдущий слайд», счётчик, опционально лазерная указка / заметки (не в MVP).
- **Состояние слайда:** синхронизировать **индекс слайда** между ведущим и зрителями (Socket.IO: например `presentation:sync` с `{ roomId, slideIndex, presenterPeerId }`).
- **Медиа:** определить, идёт ли один **video track** (захват canvas) или **DataChannel** + локальный рендер PDF на всех (меньше битрейт, больше логики на клиентах).
- **Бэкенд (если PPTX):** сервис конвертации PPTX → PDF/PNG, лимиты размера файла, антивирус/карантин — по политике продукта.
- Расширить **PHASE_A**-совместимость: в `appData` различать `screen` | `presentation` для UI и статистики.

### Риски / вопросы

- Производительность **canvas + captureStream** на слабых машинах и мобильных.
- Лицензии библиотек для PPTX.
- Кто может загрузить презентацию: только «хост» комнаты или любой участник (после авторизации).

### Критерий готовности

- Ведущий загружает PDF (MVP) или PPTX (после конвертации), листает слайды; остальные видят актуальный слайд с приемлемой задержкой и понятными контролами у ведущего.

## Рекомендуемый порядок работ

1. **Фаза 1** — главная отдельно (мало зависимостей, улучшает навигацию под всё остальное).  
2. **Фаза 2** — авторизация (влияет на joinRoom, список комнат, права на презентации).  
3. **Фаза 3** — презентации (самая тяжёлая; частично можно начать с **только PDF** и canvas + sync по сокету, без серверной конвертации PPTX).
