## Фаза 1 — Отдельное окно (страница) для главной

**Цель:** вынести «домашний» сценарий из текущего потока (сейчас при `status !== 'connected'` показывается `JoinPage`) в полноценную **главную страницу** как отдельный маршрут/экран.

### Задачи (черновик)

- Ввести **маршрутизацию** (например, `react-router`: `/` — главная, `/room/:roomId` или `/join` — вход в комнату, при необходимости редиректы).
- **Главная (`/`)**: брендинг, краткое описание, кнопка «Создать комнату» / «Войти по ссылке», возможно список недавних комнат (локально в `localStorage` — опционально).
- **Join** остаётся формой имени + комнаты + пресета, но живёт на отдельном URL или в модалке с главной — решение при реализации.
- Согласовать **историю URL** с уже используемым `replaceRoomInBrowserUrl` и соло-режимом (`?peer=`), чтобы не сломать deep link’и.

### Риски / вопросы

- Нужен ли **SSR** или достаточно SPA-only.
- Язык и контент главной (статический текст vs CMS позже).

### Критерий готовности

- Пользователь открывает сайт и видит отдельную главную; переход в комнату — осознанный шаг с понятным URL.

### URL и идентификатор комнаты (уточнение)

- **Не по «названию» в URL** — в пути только **стабильный технический id** (slug): короткий **nanoid** (8–12 символов) или **UUID**. Человекочитаемое имя комнаты (если понадобится) — отдельное поле в UI/API, не часть пути.
- **Рекомендуемая схема путей (SPA):**
  - `/` — главная: только вход в продукт («Создать комнату» / «У меня есть код»).
  - `/r/:roomId` — страница комнаты: форма имени + кнопка «Войти» (пока не подключены) и дальше текущий `RoomPage` после `join`.
- **Соло-зритель** сейчас: `?room=…&peer=…`. После перехода на путь удобно: **`/r/:roomId?peer=…`** (peer остаётся в query). В коде нужно брать `roomId` из **параметра маршрута**, а не только из `?room=`. Переходный период: поддержать и старый `/?room=&peer=` (редирект на `/r/...`).
- **«Создать комнату»:** клиент генерирует новый `roomId` (nanoid), делает `navigate('/r/' + id)` — пользователь вводит имя и жмёт «Войти» → тот же `joinRoom`, что сейчас.
- **«Войти по id»:** поле на главной → `navigate('/r/' + normalize(id))` (обрезка пробелов, нижний регистр по желанию).

### Что нужно на бэкенде (room-platform)

Сейчас фронт шлёт **`joinRoom`** с `{ roomId, name }`. Типичное поведение демо-mediasoup: **комната создаётся при первом join**, если её ещё нет.

| Сценарий | Нужны ли изменения на бэке |
|----------|----------------------------|
| «Создать комнату» = новый случайный `roomId` + первый участник делает `joinRoom` | **Часто нет** — если сервер уже создаёт комнату по `roomId` при join. |
| Гарантировать, что id **уникален** и не пересечётся с чужим случайным выбором | Либо длинный UUID/nanoid на клиенте, либо эндпоинт **`createRoom` → { roomId }** на сервере. |
| Запретить join в несуществующую комнату (только по приглашению) | **Да:** отдельное создание комнаты + `joinRoom` с ошибкой `ROOM_NOT_FOUND`. |
| Лимиты, TTL пустых комнат, модерация | Отдельная логика на сервере, контракт ошибок для фронта. |

**Итог:** минимальный фронт под «своя страница на комнату» **может обойтись без доработки бэка**, если остаётся модель «любой `roomId` при первом join создаёт комнату». Для продуктового поведения («только существующие комнаты») понадится **`createRoom`** и/или проверка существования до join.

## Фаза 2 — Авторизация

**Цель:** идентифицировать пользователя до/после входа в комнату, чтобы ограничивать доступ, персонализировать имя и (позже) хранить данные на сервере.

### Подфазы (на выбор)

1. **Минимум:** гостевой вход с отображаемым именём (как сейчас) + опционально «запомнить имя» в `localStorage` (уже частично есть идеи в проекте).
2. **Полноценный аккаунт:** регистрация / вход (email+пароль, magic link, OAuth — выбор стека).
3. **Связка с room-platform:** выдача **JWT** или сессии, передача токена в Socket.IO / joinRoom, проверка на сервере.

### Задачи (черновик)

- Выбрать **провайдера** (самописный backend, Supabase, Clerk, Keycloak и т.д.) и модель пользователя (`id`, `email`, `displayName`).
- Защищённые маршруты: главная публичная, комната — по политике (публичная ссылка vs только для залогиненных).
- UI: формы входа/регистрации, состояние «загрузка / ошибка», выход.
- Обновить **документацию** контрактов с бэкендом (какие поля уходит в `joinRoom` вместе с токеном).

### Риски / вопросы

- Публичные комнаты без аккаунта — остаются или уходят.
- GDPR / хранение PII, если email обязателен.

### Критерий готовности

- Один согласованный способ «кто я» для клиента и сервера; комнаты можно открывать с учётом правил доступа.

## Фаза 3 — Презентации (PPTX / PDF) как «демонстрация» с управлением

**Цель:** загрузить файл **PPTX** или **PDF** и показывать его участникам **как отдельный источник демонстрации** (не захват всего экрана), с **элементами управления** (листание слайдов, полноэкранный режим у зрителя, пауза и т.д. — уточняется).

### Подходы (на высоком уровне)

| Формат | Варианты реализации | Плюсы / минусы |
|--------|---------------------|----------------|
| **PDF** | Рендер страниц в `<canvas>` / `pdf.js`, трансляция кадра или отдельных страниц как видео/изображения | Проще контролировать «слайды»; нужен пайплайн в WebRTC (см. ниже). |
| **PPTX** | Конвертация на **сервере** в PDF или набор изображений; либо клиентская библиотека (тяжело, ограниченно) | Надёжнее через сервер; без бэкенда — очень ограниченно. |
| **Трансляция** | Текущий путь **screen share** заменить/дополнить **виртуальным MediaStream** (`canvas.captureStream()` + тайминг) или отдельным video track с картинками | Вписывается в mediasoup как video producer с `appData: { source: 'presentation' }`. |

### Задачи (черновик)

- **UI:** зона загрузки файла, превью, панель «следующий / предыдущий слайд», счётчик, опционально лазерная указка / заметки (не в MVP).
- **Состояние слайда:** синхронизировать **индекс слайда** между ведущим и зрителями (Socket.IO: например `presentation:sync` с `{ roomId, slideIndex, presenterPeerId }`).
- **Медиа:** определить, идёт ли один **video track** (захват canvas) или **DataChannel** + локальный рендер PDF на всех (меньше битрейт, больше логики на клиентах).
- **Бэкенд (если PPTX):** сервис конвертации PPTX → PDF/PNG, лимиты размера файла, антивирус/карантин — по политике продукта.
- Расширить **PHASE_A**-совместимость: в `appData` различать `screen` | `presentation` для UI и статистики.

### Риски / вопросы

- Производительность **canvas + captureStream** на слабых машинах и мобильных.
- Лицензии библиотек для PPTX.
- Кто может загрузить презентацию: только «хост» комнаты или любой участник (после авторизации).

### Критерий готовности

- Ведущий загружает PDF (MVP) или PPTX (после конвертации), листает слайды; остальные видят актуальный слайд с приемлемой задержкой и понятными контролами у ведущего.

## План правок — чат, меню (шевроны), раскладки, демонстрация, сеть

Отдельная **волна доработок** текущей комнаты (без привязки к фазам 2–3 выше). Можно брать блоками по приоритету.

### 1. Чат: уведомления и бейдж счётчика

**Симптомы:** не всплывают уведомления о новых сообщениях; нет (или не виден) бейдж с количеством непрочитанных.

**Направление работ:**

- Проверить цепочку **`RoomActivityNotifyRef`** в `useRoom` → `RoomSession` / `RoomPage`: при закрытом чате вызывается ли `bumpUnread`, обновляется ли `chatUnreadCount`.
- UI чата: кнопка чата в `ControlsBar` — отображение **бейджа** (число / точка), стили при `chatUnreadCount > 0`.
- «Всплывающие» уведомления: либо **краткий тост** (как у «Пригласить участников») при входящем сообщении при закрытом чате, либо компактный **preview** в углу — зафиксировать выбор при реализации.
- Убедиться, что при **открытии** чата счётчик сбрасывается (уже есть эффект в `RoomSession` — сверить с фактическим UI).

**Файлы (ориентир):** `useRoom.ts`, `RoomSession.tsx`, `RoomPage.tsx`, `ControlsBar.tsx`, `RoomChatPanel.tsx`, `index.css`.

**Статус (2026-04):** бейдж в режиме «только иконки», тост с текстом/автором при новом сообщении (не реакция), сброс превью при открытии чата.

---

### 2. Меню по шеврону: закрытие и иконка «вниз» при открытом состоянии

**Цель:**

- Закрывать выпадающие меню по **повторному нажатию на шеврон** (не только кликом «в пустоту»).
- Когда меню **открыто**, шеврон показывать **стрелкой вниз** (сейчас визуально «вверх» — поменять логику/класс на противоположное при `open`).

**Направление работ:**

- Найти все места с **chevron** + popover/dropdown (например пресет, экран, устройства, чат) в `ControlsBar` и связанных компонентах.
- Паттерн: `onClick` на **кнопке-шевроне** toggles `open`; `stopPropagation` где нужно; внешний клик (backdrop) оставить как дополнительный способ закрытия.
- CSS: класс модификатор для открытого состояния (`transform: rotate(...)` или смена `svg`/символа).

**Файлы (ориентир):** `ControlsBar.tsx`, возможно вложенные меню SRT/настроек, `index.css`.

**Статус (2026-04):** `shouldClosePopoverOnOutsidePointer` — клик по шеврону/кнопке той же `ctrl-group` не закрывает поповер на `mousedown` до `click`; шеврон при открытом меню смотрит **вниз** (`.ctrl-chevron__glyph` + `--open`).

### 3. Виды (layout): убрать «масштаб видео», добавить режим «говорящий» (как Google Meet)

**Цель:**

- Убрать из UI опцию **«масштаб видео»** (object-fit cover/contain и связанный переключатель).
- Добавить режим **«говорящий»**: определение **активного говорящего** по уровню аудио (локальный + удалённые треки), вывод его на **большую область**, остальные — в **горизонтальной полосе** превью (аналог stage + filmstrip в Meet).

**Направление работ:**

- **Детекция речи:** Web Audio API (`AnalyserNode`) на playout-треках удалённых и на локальном микрофоне (или существующий `AudioMeter` / метрики — переиспользовать идеи). Порог, сглаживание, таймаут «тишина», чтобы не дёргать раскладку на шум.
- **Новый `LayoutMode`** (например `active_speaker` / переименование существующего `meet`/`speaker` — уточнить, не дублировать текущие `speaker`/`meet`).
- **RoomPage:** одна крупная плитка = текущий «speaker», полоса снизу/сверху — остальные участники (как уже частично сделано для meet/speaker — свести к одному понятному режиму).
- Удалить переключатель и состояние **objectFit** из шапки/настроек и связанные пропсы там, где они только для «масштаба».

**Риски:** нагрузка CPU от нескольких `AnalyserNode`; задержка переключения; корректность при **mute** и при **screen share** (кто в фокусе — говорящий или демонстрирующий — правило продукта).

**Файлы (ориентир):** `RoomPage.tsx`, `ParticipantCard.tsx`, `AudioMeter.tsx`, `ControlsBar.tsx`, при необходимости новый хук `useActiveSpeaker.ts`, `index.css`.

### 4. Демонстрация экрана: возврат к камере, согласование с режимами «демонстрация» и «спикер»

**Цель:**

- Стабильно **останавливать** демонстрацию и **возвращаться** к обычной камере (треки, producer’ы, UI плиток).
- Явно развести смыслы: режим раскладки **«спикер»** (кто на сцене) vs источник видео **«демонстрация экрана»** (отдельный producer / плитка экрана), чтобы не ломать друг друга при переключениях.

**Направление работ:**

- Пройти сценарии: старт screen share → стоп → камера снова в эфире; смена layout во время screen share; пин/спикер при наличии `screenStream`.
- `useRoom`: `stopScreenShare`, `toggleScreenShare`, закрытие `screenProducer`, обновление `localScreenStream` и карты участников — проверить гонки и `producerClosed` с бэка.
- UI: кнопка «Завершить демонстрацию», диалоги (`ConfirmDialog`), соответствие **режиму спикера** из п.3 (если экран в фокусе — не перетирать говорящим без правил).

**Файлы (ориентир):** `useRoom.ts`, `RoomPage.tsx`, `LocalScreenShareTile.tsx`, `ControlsBar.tsx`, `ParticipantCard.tsx`.

### 5. Индикация качества интернета (со стороны клиента)

**Цель:** показать пользователю **понятный сигнал** о стабильности связи (как минимум «хорошо / средне / плохо»), без обязательной доработки бэка на первом шаге.

**Направление работ:**

- **Источник метрик (предпочтительно WebRTC):** периодический опрос **`getStats()`** у `RTCPeerConnection`, к которым привязаны send/recv transport’ы mediasoup-client (доступ через объекты транспортов/`handler` — уточнить в актуальной версии API). Интересуют, например: **packet loss** (вход/исход), **RTT**, **jitter**, при возможности **доступный исходящий битрейт** / адаптация отправки.
- **Агрегация:** сглаживание за окно 2–5 с, пороги для уровней **good / fair / poor** (подобрать эмпирически; разные пороги для Wi‑Fi vs эталон не нужны на MVP).
- **UI:** компактный индикатор в **шапке комнаты** или рядом с контролами (иконка «сигнал» / три полоски / точки), **tooltip** или раскрывающаяся подсказка с краткими цифрами для продвинутых пользователей.
- **Опционально позже:** лёгкий **HTTP ping** до signaling (`/api/...` или отдельный `HEAD`) как дополнение к WebRTC (отдельно измеряет «доступность хоста», не заменяет качество медиа-пути).
- **Ограничения:** метрики отражают **этого клиента**, не «качество комнаты целиком»; при отсутствии подключения показывать «нет данных» / скрывать.

**Риски:** различия браузеров в полях `getStats()`; нагрузка от частого опроса (достаточно 1–2 раза в секунду).

**Файлы (ориентир):** новый хук `useConnectionQuality.ts` (или логика внутри `useRoom.ts`), `RoomPage.tsx` / `ControlsBar.tsx`, `index.css`.

### Предлагаемый порядок внутри этой волны

1. **Чат (бейдж + уведомления)** — изолированно, быстрый пользовательский эффект.  
2. **Шевроны** — мелкая правка UX.  
3. **Демонстрация / камера** — стабилизация медиа до усложнения раскладок.  
4. **Режим говорящего + убрать масштаб** — больше всего кода и решений по продукту.  
5. **Индикация качества сети** — в основном клиент + `getStats`, можно параллелить с п.2–3 после прототипа метрик.

## Рекомендуемый порядок работ

1. **Фаза 1** — главная отдельно (мало зависимостей, улучшает навигацию под всё остальное).  
2. **Фаза 2** — авторизация (влияет на joinRoom, список комнат, права на презентации).  
3. **Фаза 3** — презентации (самая тяжёлая; частично можно начать с **только PDF** и canvas + sync по сокету, без серверной конвертации PPTX).
