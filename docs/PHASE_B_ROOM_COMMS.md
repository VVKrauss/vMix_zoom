# Фаза B: чат и реакции в комнате (Socket.IO)

Контракт для **room-platform**: ретрансляция внутри `roomId`, лимиты длины/частоты и (позже) модерация — на стороне сервера.

## События

### `chat:message` (клиент → сервер)

Тело:

```json
{
  "roomId": "string",
  "text": "string"
}
```

- `text` после `trim`, не пустой; на фронте обрезка **2000** символов до отправки.
- Сервер подставляет **`peerId`** (socket.id), **`name`** (из joinRoom или профиля), **`ts`** (unix ms), валидирует и рассылает в комнату тем же событием.

### `chat:message` (сервер → клиенты в комнате)

```json
{
  "roomId": "string",
  "peerId": "string",
  "name": "string",
  "text": "string",
  "ts": 1730000000000
}
```

Опционально в ответе **`joinRoom`** (ack): поле **`chatHistory`** — массив последних N сообщений того же формата (без `roomId` или с ним — как договоритесь; фронт принимает оба).

### `reaction` (клиент → сервер)

```json
{
  "roomId": "string",
  "emoji": "👍",
  "ttlMs": 3500
}
```

- `emoji` — whitelist из 5 символов (фронт и сервер совпадают): **👍 👏 ❤️ 😂 🔥**.
- `ttlMs` опционально (по умолчанию на сервере ~3000–4000).

### `reaction` (сервер → клиенты в комнате)

```json
{
  "roomId": "string",
  "peerId": "string",
  "emoji": "👍",
  "ts": 1730000000000,
  "ttlMs": 3500
}
```

Фронт показывает анимацию на плитке участника `peerId` в течение `ttlMs`. Персистентности нет.

## Рекомендации бэкенду

- Лимит длины текста и **rate limit** на `chat:message` / `reaction` на сокет или `peerId`.
- Без авторизации имена берутся из `joinRoom`; при спаме — отключение сокета или бан `peerId`.
- Системные сообщения и «печатает» — отдельные события, не в минимальном протоколе.

## Фронт (этот репозиторий)

- Слушает и шлёт события выше; при отсутствии поддержки на сервере чат/реакции просто не обновляются.
- История только в памяти вкладки + опционально `chatHistory` из `joinRoom`.
