# Аудио-источники (mic + screen-audio) для микшера на клиенте

## Статус

В текущей версии проекта **screen-audio микшер удалён**: демонстрация экрана работает как **video-only** и не создаёт/не подменяет аудио producer’ы.

Цель: поддержать **вариант A** — у одного участника одновременно могут быть **2 независимых аудио producer**:

- **mic** — микрофон участника
- **screen** — звук демонстрации экрана/вкладки (getDisplayMedia audio)

Это нужно, чтобы на клиентах (комната / solo-ссылка) был управляемый микшер: отдельно `Mic` и отдельно `Screen`.

---

## 1) Маркировка producer на сервере

Для каждого producer сервер должен пробрасывать в дескриптор, который уходит в:

- ack `joinRoom` (`existingProducers`)
- событие `newProducer`

### Рекомендуемый контракт

Добавить в `ProducerDescriptor` поле:

- `audioSource?: 'mic' | 'screen'` (только для `kind: 'audio'`)

и/или пробрасывать `appData.source` в дескриптор (как уже делается для видео), где для screen-audio будет один из вариантов:

- `appData.source = 'screen_audio'` (предпочтительно, явнее)
- или `appData.source = 'screen'` при `kind: 'audio'`

### Привязка к «отдельному источнику» экрана

Если в системе экранное видео публикуется как отдельный виртуальный `peerId` (см. `ownerPeerId`), то для screen-audio нужно:

- публиковать audio producer с тем же виртуальным `peerId` (чтобы экранное видео и экранный звук совпадали по `peerId`)
- указывать `ownerPeerId` = peerId владельца (камеры), чтобы UI мог связать источник с человеком

---

## 2) Поведение при включении шаринга со звуком

Требование: **микрофон не выключается** при включении screen-audio.

Правильная реализация (вариант A):

- не делать `replaceTrack()` у микрофонного producer
- вместо этого создать **второй** audio producer для `displayAudio` и пометить его как `audioSource='screen'`

---

## 3) producerClosed / cleanup

Событие `producerClosed` должно корректно закрывать любой из audio producer’ов.
Клиенты после закрытия удаляют только соответствующий источник (mic/screen), не трогая второй.
