# Production cutover (минимальный даунтайм)

## За 24–48 часов

- [ ] Повторная репетиция по [`MIGRATION_CUTOVER_STAGING.md`](./MIGRATION_CUTOVER_STAGING.md).
- [ ] Бэкап облачного проекта: свежий `pg_dump` + экспорт списка `storage.objects` (SQL или API).
- [ ] Окно обслуживания согласовано (минуты).

## Freeze writes (опционально, но снижает риск расхождения)

Варианты:

1. **Фича-флаг** во фронте: блокировать отправку сообщений / загрузки с баннером «техработы».
2. **Только ночное окно** без freeze, если трафик низкий — тогда финальный `pg_dump` должен быть максимально свежим.

## Финальная синхронизация данных

1. Повторить `pg_dump` с облака (или инкремент, если настроена репликация).
2. Коротко остановить запись в облако (freeze).
3. Залить финальный дамп на VPS (`scp`) и выполнить restore скриптом [`scripts/vps/db-restore.sh`](../scripts/vps/db-restore.sh).
4. Догнать Storage: перенести объекты в S3 (скрипт `migrate:storage` удалён как legacy).

## Переключение трафика

1. Обновить DNS / CDN на API VPS (Kong) **или** обновить `VITE_SUPABASE_URL` в новой сборке фронта и задеплоить статику.
2. Если используется `VITE_SUPABASE_PROXY_ORIGIN` — переключить прокси на тот же новый API.
3. Снять freeze writes.

## Сразу после

- [ ] Smoke: логин, одно ЛС, один канал, один upload, один realtime канал.
- [ ] Метрики: 5xx на `/rest/v1`, ошибки Realtime websocket.
- [ ] Логи: `docker compose logs -f --tail=200` по сервисам `kong`, `auth`, `rest`, `realtime`, `storage`, `functions`.

## Откат

Пока freeze активен, облако остаётся источником истины.

1. Вернуть DNS / переменные окружения на облачный `*.supabase.co`.
2. Снять freeze на облаке (если был).
3. Разбор инцидента на копии VPS БД (не трогать прод облако без бэкапа).

## Пост-миграция

- Мониторинг 24–72 ч.
- Отключить запись в облачный проект (read-only) после стабилизации.
- Обновить [`docs/DATABASE.md`](./DATABASE.md) ссылкой на фактический endpoint VPS (по желанию).
