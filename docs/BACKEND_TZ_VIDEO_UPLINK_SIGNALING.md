# ТЗ: сигналинг «качество исходящего видео → SFU» (`reportVideoUplink` / `videoUplink`)

## 1. Цель

В комнате на карточке каждого участника показывается **качество исходящего видео этого участника до SFU** (не качество «между двумя пользователями» и не downlink зрителя). Для **своей** карточки клиент считает метрики локально (`getStats` на video producer). Для **чужих** карточек каждый клиент периодически шлёт снимок на сервер; сервер **рассылает** его остальным участникам комнаты.

Без серверного relay чужие индикаторы **не работают**.

## 2. Транспорт

Socket.IO (тот же сокет, что `joinRoom`, `newProducer`, `chat:message` и т.д.).

## 3. Входящее событие от клиента: `reportVideoUplink`

### 3.1. Когда шлёт клиент

Только у пользователя с **включённой камерой** (video producer типа camera/vmix, не screen/studio). Не чаще чем **раз в 2 секунды** на клиенте (throttle на фронте).

### 3.2. Тело (payload)

Объект JSON. Имена полей **как ниже** (camelCase); опционально можно принимать snake_case как синонимы при парсинге — на фронте при приёме **чужого** broadcast они не используются, только ниже перечисленные.


| Поле           | Тип          | Обязательно | Описание                                                                                                                                                                                                   |
| -------------- | ------------ | ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `roomId`       | string       | да          | ID комнаты (как в остальных событиях).                                                                                                                                                                     |
| `bitrateBps`   | number       | да          | Оценка битрейта исходящего видео, бит/с.                                                                                                                                                                   |
| `fractionLost` | number       | да          | Доля потерь пакетов за интервал, 0…1.                                                                                                                                                                      |
| `jitterMs`     | number       | null        | нет                                                                                                                                                                                                        |
| `level`        | number (1…5) | нет         | Сырой уровень с клиента; **фронт при приёме пересчитывает** из `bitrateBps` + `fractionLost` + `jitterMs`, поле можно не слать или слать — на отображение для других клиентов достаточно трёх метрик выше. |


**Важно:** идентификатор отправителя (`peerId`) **не** должен браться из payload клиента без проверки. Нужно подставлять **авторитетный** peer/socket id участника из сессии комнаты (как для других событий).

### 3.3. Обработка на сервере

1. Проверить, что сокет **вошёл в комнату** `roomId` и комната существует.
2. Определить `peerId` отправителя из **внутренней карты** «socket ↔ участник комнаты» (или эквивалент), **не доверять** произвольному `peerId` из тела.
3. Валидация чисел: `bitrateBps` и `fractionLost` должны быть конечными числами; при мусоре — игнорировать событие (или ответить ошибкой, если используете ack).
4. **Rate limit** (рекомендация): не более **1 сообщения в 1–2 с на участника в комнату** для защиты от спама (фронт уже шлёт ~2 с).
5. Рассылка — п. 4.

## 4. Исходящее событие к клиентам: `videoUplink`

### 4.1. Кому слать

Всем сокетам, состоящим в **той же комнате** `roomId`, **кроме** отправителя (опционально можно слать и отправителю — фронт отфильтрует по `peerId === socket.id`, но проще не слать себе).

### 4.2. Тело (payload)

Минимум для совместимости с фронтом (`parseVideoUplinkBroadcast` в `useRoom.ts`):


| Поле           | Тип    | Обязательно | Описание                                                |
| -------------- | ------ | ----------- | ------------------------------------------------------- |
| `roomId`       | string | да          | Та же комната.                                          |
| `peerId`       | string | да          | **Кто** снял метрику (отправитель `reportVideoUplink`). |
| `bitrateBps`   | number | да          |                                                         |
| `fractionLost` | number | да          |                                                         |
| `jitterMs`     | number | null        | нет                                                     |


Фронт дополнительно проверяет: `roomId` совпадает с текущей комнатой; `peerId` не равен локальному `socket.id` (свой пакет не показываем из broadcast — локальные метрики считаются через `getStats`).

**Snake_case на приёме:** клиент при разборе допускает `bitrate_bps`, `fraction_lost`, `jitter_ms` — при желании сервер может дублировать поля, но **достаточно camelCase**, как в таблице.

## 5. Семантика для продукта

- Показываем **аплинк участника к SFU** (путь «браузер → сервер медиа» для его видеопотока).
- Это **не** «качество связи между двумя людьми» и **не** «как ко мне приходит его картинка» (это был бы consumer/downlink зрителя).

## 6. Критерии приёмки

1. Два клиента в одной комнате с камерами: у **A** на карточке **B** появляются полоски/данные в оверлее (после стабилизации ~2–5 с).
2. В DevTools → WS видно: от каждого клиента уходят `reportVideoUplink`, другим приходят `videoUplink` с корректным `peerId`.
3. Участник без камеры **не** обязан слать `reportVideoUplink`; индикатор у других для него может отсутствовать (текущий фронт так и задуман).
4. Подмена `peerId` с клиента в `reportVideoUplink` **не** приводит к подмене чужого индикатора (сервер подставляет свой id).

## 7. Ссылка на фронт (репозиторий клиента)

- Отправка: `src/hooks/useRoom.ts` — `socket.emit('reportVideoUplink', { roomId, level, bitrateBps, fractionLost, jitterMs })` (throttle ~2 с).
- Приём: тот же файл — `socket.on('videoUplink', …)` → `parseVideoUplinkBroadcast`, обновление `peerUplinkVideoQualityRef`.
- Разбор метрик локально: `src/utils/uplinkVideoStats.ts`, агрегирование уровня: `src/utils/inboundVideoStats.ts` (`buildQuality`).
