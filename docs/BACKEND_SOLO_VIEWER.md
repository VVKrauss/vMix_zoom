# Бэкенд: ссылки с `?room=`, соло-просмотр (`?peer=`) и Socket.IO

Документ для **room-platform** (mediasoup + Socket.IO). Описывает, как фронт использует query-параметры и какое событие нужно добавить для страницы «только один участник».

---

## 0. Кратко

| URL | Поведение фронта | Нужен ли новый API на бэке |
|-----|------------------|----------------------------|
| `/?room=ИД` | Экран входа с подставленной комнатой; после входа/выхода URL синхронизируется | **Нет** — обычный `joinRoom` |
| `/?room=ИД&peer=ИД` | Соло-просмотр: только видео+аудио выбранного пира, без камеры | **Да** — `joinRoomAsViewer` |

**Важно:** в query параметр **`peer`** на фронте передаётся в сокет как **`watchPeerId`** и должен совпадать с **`peerId` в продюсерах** = **`socket.id`** того участника в момент, когда ссылку скопировали.

---

## 1. Соглашения по URL (фронт)

Бэкенду **не нужно** разбирать эти URL на сервере — это чисто клиентская навигация. Имеет смысл знать формат для поддержки и тестов.

### 1.1. Только комната

```text
https://redflow.online/?room=test
http://localhost:5173/?room=test
```

- Подставляется поле «Комната» на экране входа.
- При изменении поля «Комната» фронт обновляет `?room=` через `history.replaceState`.
- После успешного **«Войти»** в URL остаётся актуальная `room`, параметр **`peer` сбрасывается** (чтобы не оказаться в соло-режиме по ошибке).
- После **«Выйти»** из комнаты снова показывается вход, в URL сохраняется **`?room=…`** последней комнаты — удобно зайти повторно.

### 1.2. Комната + участник (соло-страница)

```text
https://redflow.online/?room=test&peer=SOCKET_ID
```

- Если есть **оба** параметра, фронт **не** показывает обычный join, а сразу подключается как зритель (см. раздел 3).
- Как выглядит соло-страница (полный экран, звук, Escape, клик, несколько вкладок) — см. **[FRONTEND_SOLO_VIEWER.md](./FRONTEND_SOLO_VIEWER.md)**.
- При выходе с соло-страницы из URL удаляется только **`peer`**, **`room` сохраняется** — открывается обычный вход в ту же комнату.

### 1.3. Откуда пользователь берёт ссылку

Фронт копирует **полный абсолютный URL** (origin + pathname + query):

1. **Инфо-оверлей** на плитке (если включён «Инфо»): кнопка **«Соло-страница»**.
2. **Контекстное меню** по плитке (ПКМ или долгое нажатие): пункт **«Копировать полный URL (?room=…&peer=…)»**.
3. Там же в меню по-прежнему: копирование **порта SRT** и **строки SRT** (`connectUrlPublic`), когда сессия SRT уже есть.

Меню открывается, если есть **хотя бы одно** из: порт, SRT-строка, **или** известны `roomId` и `peerId` плитки (тогда доступен только пункт соло-URL).

## 2. Идентификаторы: что должно совпадать

- **`room`** в URL = **`roomId`** во всех событиях (`joinRoom`, `createWebRtcTransport`, `consume`, …).
- **`peer`** в URL = **`watchPeerId`** в `joinRoomAsViewer` = **`peerId`** в объектах продюсеров (`existingProducers`, `newProducer`, ответы API), которые относятся к этому человеку.

Рекомендация: везде для «логического участника в комнате» в сигналинге использовать **текущий `socket.id`** как `peerId`, чтобы ссылки и инфо-оверлей не расходились с сервером.

После **переподключения** участника `socket.id` меняется — старая ссылка `?peer=…` перестаёт указывать на него; это ожидаемо.

---

## 3. Обычный участник (без изменений)

Поток как раньше:

- `joinRoom` → `rtpCapabilities`, `existingProducers`
- `createWebRtcTransport` (send + recv)
- `connectTransport`, `produce`, `consume`, `resumeConsumer`
- события `newProducer`, `peerLeft`, `producerClosed` (если используете), `srtStarted`, и т.д.

Ничего из этого для работы **только** `?room=` менять не нужно.

---

## 4. Соло-зритель: событие `joinRoomAsViewer`

Фронт при `?room=` + `?peer=` **не** вызывает `getUserMedia`, **не** создаёт send-transport и **не** вызывает `produce`.

### 4.1. Запрос (emit)

```json
{
  "roomId": "string",
  "watchPeerId": "string"
}
```

`watchPeerId` — значение query-параметра **`peer`**.

### 4.2. Ответ (обязательный Socket.IO ack)

Как у `joinRoom`: последний аргумент — функция `callback(...)`.

**Успех:**

```json
{
  "rtpCapabilities": { },
  "producers": [
    {
      "producerId": "...",
      "peerId": "<watchPeerId>",
      "kind": "audio",
      "name": "Имя участника"
    },
    {
      "producerId": "...",
      "peerId": "<watchPeerId>",
      "kind": "video",
      "name": "Имя участника"
    }
  ]
}
```

- **`rtpCapabilities`** — тот же объект, что отдаёте в `joinRoom` (для `mediasoup-client` `Device.load()`).
- **`producers`** — только продюсеры участника с `peerId === watchPeerId` (audio + video), в **том же формате**, что элементы `existingProducers` при `joinRoom`. Если треков ещё нет — **`[]`**.

**Ошибка:**

```json
{
  "error": "человекочитаемое сообщение"
}
```

### 4.3. Таймаут на фронте

Если ack **не пришёл** в течение **20 секунд**, клиент показывает ошибку таймаута (часто значит, что обработчик на сервере ещё не реализован).

### 4.4. Дальнейшие шаги на клиенте после успешного ack

1. `createWebRtcTransport` с `{ roomId }` — ожидается **recv**-транспорт (как у обычного клиента).
2. `connectTransport` с `dtlsParameters`.
3. Для каждого элемента `producers` — `consume` + **`resumeConsumer`** с **обязательным** ack-колбэком (пустая функция с клиента допустима, но аргумент колбэка на сервере должен быть вызван — иначе типичная ошибка `callback is not a function`).
4. Подписка на **`newProducer`**: клиент обрабатывает только записи с `producer.peerId === watchPeerId`.
5. **`peerLeft`**: если `peerId === watchPeerId`, клиент показывает «участник вышел» и рвёт соединение.

**Send transport и `produce` для зрителя не нужны.**

---

## 5. Поведение сервера (рекомендации)

### 5.1. Комната Socket.IO / рассылка событий

Сокет зрителя должен получать те же события комнаты, что нужны для приёма:

- **`newProducer`** — рассылать всем в комнате **включая зрителей**; фронт отфильтрует по `peerId`.
- **`peerLeft`** — при выходе целевого участника.

### 5.2. Видимость зрителя для остальных

На выбор продукта:

- **A:** не считать зрителя «гостем» в UI (не слать `peerJoined` о зрителе, не увеличивать счётчик людей).
- **B:** тип `role: 'viewer'` — явно не создавать producers и не требовать медиа.

Главное — не ломать существующий `joinRoom` и не требовать от зрителя публикации треков.

### 5.3. Целевой участник ещё не в комнате или ещё без треков

Рекомендуется вернуть **успех** с `producers: []`, а не ошибку, если комната существует и `watchPeerId` может появиться позже. Тогда клиент дождётся **`newProducer`**.

Если такого пира в комнате **никогда** не было, можно вернуть **`error`** — фронт покажет сообщение.

### 5.4. Отключение зрителя

Обычно достаточно `disconnect` сокета: на сервере закрыть transports/consumers зрителя и убрать его из комнаты.

Отдельное событие `leaveRoom` от зрителя **не обязательно**, если уже чистите ресурсы по `disconnect`.

---

## 6. REST `/api/frontend/*`

Соло-страница **не** использует polling комнаты для входа; достаточно сокета и `joinRoomAsViewer`.

Эндпоинты вроде `GET /api/frontend/rooms/:roomId` по-прежнему могут быть полезны для дашборда и не конфликтуют с зрителем.

---

## 7. SRT и egress

Зритель **не** создаёт producers → ваши сценарии **SRT / FFmpeg / egress** от обычных участников **не меняются**.

Строка **`connectUrlPublic`** и порт по-прежнему отдаются тем, кто публикует аудио+видео; зрителю это не требуется.

---

## 8. CORS и разнесённые домены

Если статика на **`https://redflow.online`**, а API/WebSocket на **`https://s.redflow.online`**, в CORS Socket.IO и HTTP нужно разрешить origin статики. Соло-страница использует тот же **`VITE_SIGNALING_URL`**, что и обычная комната.

---

## 9. Безопасность

Сейчас ссылка `?room=&peer=` по сути **публичная**: зная комнату и актуальный `socket.id`, можно открыть поток.

Идеи на будущее (не в текущем фронте):

- подписанный токен в query вместо сырого `peer`;
- короткий `viewerSlug` в БД с привязкой к комнате и участнику;
- ограничения на nginx / rate limit.

---

## 10. Чеклист для бэкенда

1. [ ] Реализовать **`joinRoomAsViewer`** с ack: при успехе — `rtpCapabilities` + `producers` (только `watchPeerId`, или `[]`).
2. [ ] При ошибке — `{ error: "..." }` в ack.
3. [ ] Подключать сокет зрителя к комнате для **`newProducer`** и **`peerLeft`**.
4. [ ] Не требовать send-transport / **`produce`** от зрителя.
5. [ ] Один вызов **`createWebRtcTransport`** (recv) → **`connectTransport`** → **`consume`** → **`resumeConsumer`** (с ack).
6. [ ] При выходе целевого участника — **`peerLeft`** с его `peerId`.

После этого работают:

- прямые ссылки `https://ваш-домен/?room=…&peer=…`;
- кнопки копирования этой ссылки в UI.

---

## 11. Справка: что на бэке **не** нужно делать

- Парсить или редиректить URL с `?room=` / `?peer=` — этим занимается только браузер и фронт.
- Менять формат **`joinRoom`** / **`produce`** / **`consume`** для обычных гостей из-за соло-режима.

При вопросах по полям ответов сверяйтесь с текущими типами продюсеров в вашем `joinRoom` — соло-клиент ожидает **тот же** shape объектов в `producers` и в `newProducer`.
